# Technical Specification Draft: Multi-Location Asterisk PBX Config Portal

## 1. Overview

Build a central, WARP-only PBX management portal for a company group with multiple locations connected over Cloudflare WARP. The portal manages locations, extensions, Cisco phone provisioning, trunks, IVRs, DIDs, queues, voicemail, recordings, dashboards, versioned per-location config exports, and deployment to remote PBX servers.

Each location runs its own local PBX. Phones register only to their local PBX. Inter-location calling is routed between PBXs over WARP.

The portal generates a versioned per-location config bundle containing deployment files plus two separated config folders:

```text
location-export.zip
  docker-compose.yml
  .env.example
  asterisk/
  tftp/
```

The portal can deploy a selected config version to a remote Debian Docker server over SSH, update the configured host volumes, and reload Asterisk after admin confirmation.

## 2. Core Decisions

- Central portal: Django + HTMX custom UI, Ant Design style.
- Database: PostgreSQL.
- Hosting: Docker Compose on an internal management server reachable only through LAN or Cloudflare WARP.
- Authentication: named users with role-based access and audit logging.
- PBX runtime: Asterisk 22 LTS in a pinned custom Docker image.
- PBX host OS target: Ubuntu/Debian Linux.
- Asterisk SIP stack: PJSIP only.
- SIP transport: TCP enabled and generated for Cisco phone registration.
- Phone models: Cisco CP-9971 default, selectable CP-9951 and CP-8961.
- Cisco provisioning: per-MAC `SEP<MAC>.cnf.xml` files.
- Inter-location topology: Via PBX.
- Inter-location default trunk protocol: IAX2, with SIP trunk support also available.
- Extension numbering: globally unique 4-digit extensions.
- Device auth: shared credentials per extension.
- Registration limit: up to 5 registered device per extension.
- Export validation: warnings allowed, except missing emergency route/caller ID blocks export unless the affected extension has an explicit admin-set 911 disable override.
- Configuration lifecycle: every export is versioned, can be rolled back, and is reported back by the PBX agent as the active deployed config version.

## 3. Users and Roles

The portal should support these roles:

- Viewer: can view locations, dashboards, and generated config metadata.
- Editor: can manage configuration and export location bundles.
- Operator: can use live operational controls.
- Admin: can manage users, roles, system settings, and all data.

All config changes, exports, and live PBX actions must be audit logged.

## 4. Location Model

Each location should include:

- Name and short code.
- Local LAN subnet.
- PBX host LAN IP via DHCP.
- PBX WARP IP via DHCP (assigned by cloudflare).
- Default DID.
- Emergency caller ID.
- Emergency-capable outbound trunk.
- Remote deployment host, SSH port, SSH username, and authorized portal SSH key.
- SIP/RTP/IAX-related network settings.
- Recording retention setting, default 90 days.
- Optional SMTP settings for voicemail email.
- Agent authentication token/secret.
- AMI credentials generated by the portal.
- Current active config version as reported by the PBX.
- Last deployed config version and deployment status as recorded by the portal.

Assumption: each location has a unique, non-overlapping subnet within the group private address space.

## 5. Extension Model

Each extension should include:

- Globally unique 4-digit extension number.
- Display name.
- Location.
- Shared SIP username/secret.
- Optional direct DID.
- Optional voicemail PIN.
- Optional voicemail email.
- Caller ID behavior.
- Recording policy.
- Optional force-disable 911 route flag.
- Queue/ring group membership.
- Paging group membership.
- Assigned Cisco phone line appearances.

Generated PJSIP behavior:

- One endpoint/auth/AOR per extension.
- `max_contacts=5`.
- TCP transport enabled.
- Shared extension credentials.
- Every registered line appearance counts toward the 5-contact limit.
- If `force_disable_911` is enabled for an extension, emergency dialing from that extension is blocked in generated dialplan and must be prominently warned and audit logged.

## 6. Cisco Phone Provisioning

Supported models:

- Cisco CP-9971, default.
- Cisco CP-9951.
- Cisco CP-8961.

Provisioning requirements:

- Generate one `SEP<MAC>.cnf.xml` file per phone.
- Normalize and validate MAC addresses.
- Use Cisco SIP XML structure based on the referenced `SEPMAC.cnf.xml` guidance.
- Set `transportLayerProtocol` to TCP.
- Support multiple registered lines per phone.
- Support plain speed-dial entries for the expansion/side screen.
- Do not bundle Cisco firmware files.
- Generate firmware/load placeholders and checklist files in the TFTP folder.
- DHCP option 66 is configured outside this system.

Phone book:

- Generate a global company directory containing all active extensions.
- Include location labels/grouping.
- Serve directory XML through a small local HTTP static service.
- Cisco phone configs should point `directoryURL` to the local HTTP service.

## 7. PBX Deployment Bundle

Each per-location config version should contain:

```text
docker-compose.yml
.env.example
asterisk/
tftp/
```

The per-location Docker Compose stack should include:

- Asterisk 22 LTS custom image.
- TFTP service for Cisco provisioning.
- HTTP static service for phone book/provisioning assets.
- PBX agent sidecar.
- Host networking for Asterisk.

Deployment behavior:

- The portal stores or references SSH key pairs for each remote Debian PBX host.
- Only authorized admins can configure SSH credentials and deployment targets.
- When an admin selects "deploy", the portal connects to the remote PBX server over SSH.
- The portal uploads the selected version's `asterisk/` and `tftp/` folders to a staging directory.
- The portal verifies the staged bundle, swaps it into the configured host volume paths, and reloads Asterisk.
- Rollback redeploys a previous recorded config version and reloads Asterisk.
- Deployment status, operator, timestamp, target host, selected version, and reload result must be audit logged.

Asterisk generated config should include, at minimum:

- `pjsip.conf`
- `iax.conf`
- `extensions.conf`
- `voicemail.conf`
- `queues.conf`
- `manager.conf`
- `features.conf`
- `musiconhold.conf`
- CDR/CEL config
- Recording config
- Retention job config or helper script references

## 8. Inter-Location Calling

Default protocol:

- IAX2 for PBX-to-PBX trunks over WARP.

SIP trunk support:

- Available as an alternative per configured trunk.

Routing behavior:

- Since extensions are globally unique, each PBX should route remote extensions to the owning location's PBX.
- All phones register locally only.
- Inter-location trunks should use known WARP PBX addresses.
- Generated ACLs should allow:
  - Local phone subnet.
  - Known WARP PBX peers.
  - Configured provider endpoints.

## 9. Provider Trunks and Outbound Routing

Provider support:

- Multiple outbound VoIP providers expected.
- Providers may use SIP or IAX2.
- Credentials are stored in plaintext PostgreSQL.
- Exported Asterisk configs contain plaintext provider credentials.

Outbound routing:

- Admins define dial patterns.
- Each route has ordered provider trunks for priority/fallback.
- Caller ID selection:
  - Use extension DID if assigned.
  - Otherwise use location default DID.
  - Emergency calls use emergency caller ID.

Emergency calling:

- Each location must define an emergency-capable trunk and caller ID.
- Missing emergency route/caller ID is the only hard export block for extensions that are allowed to dial emergency routes.
- Admins may force-disable 911 routing for a specific extension.
- Force-disabling 911 for an extension must require admin permission, display a strong warning, and create an audit log entry.

## 10. Inbound DID Routing

Inbound DID routing policy:

1. If DID is assigned to a specific extension, route directly to that extension.
2. Otherwise, route to the location default destination.
3. Default destination may be IVR, ring group, queue, or extension.

Each location must support:

- Default DID.
- Optional per-extension DID.
- Default inbound destination.

## 11. IVR, Queues, Ring Groups, and Paging

IVR v1 must support:

- Uploaded prompts.
- Prompt conversion to Asterisk-friendly audio formats.
- Digit routes.
- Timeout route.
- Invalid-input route.
- Business-hours and after-hours behavior.

Ring groups:

- Named groups.
- Simultaneous or ordered ringing.
- Usable as IVR/DID destinations.

Queues:

- Static queues only.
- Admin-configured members.
- Strategy.
- Timeout.
- Retry.
- Music-on-hold.
- Overflow destination.
- No dynamic agent login/logout in v1.

Paging:

- Per-location paging groups.
- No cross-location paging by default.

Feature codes:

- Voicemail access.
- Directed pickup.
- Call parking.
- Attended/blind transfer support.
- Paging.

## 12. Voicemail

Voicemail support:

- Basic voicemail per extension.
- Optional voicemail-to-email.
- SMTP settings configurable per location.
- Voicemail should work by phone even if SMTP is not configured.
- Cisco message waiting behavior should be supported where practical.

## 13. Call Recording

Recording scope:

- Policy-based call recording, default off.
- Policies may apply per extension, queue, or route.

Storage:

- Recordings remain on each local PBX host volume.
- Default retention: 90 days.
- Retention configurable per location.

Portal behavior:

- Portal shows recording metadata and playback links through the PBX agent.
- Recording access must be role-gated and audited.

## 14. PBX Agent and Dashboard

Each location runs a PBX agent sidecar.

Agent connection:

- Agent opens an outbound authenticated WebSocket to the central portal over WARP.
- No inbound agent management port is required.

Agent responsibilities:

- Read local AMI state.
- Read CDR/CEL/call history as available.
- Report phone registration status.
- Report trunk status.
- Report active calls.
- Report queue status.
- Report recording metadata.
- Report current active config version from the deployed PBX config/version marker.
- Execute live operational commands from authorized portal users.

Live controls:

- Commands go through the PBX agent.
- Agent talks to local Asterisk AMI.
- All live actions are role-gated and audit logged.

Dashboard should show:

- Location health.
- Extension registrations.
- Trunk status.
- Active calls.
- Queue status.
- Recent calls.
- Recording availability.
- Agent connection state.
- Active PBX config version versus latest exported version.
- Deployment and rollback history.

## 15. Portal UI Requirements

Use a custom Django + HTMX interface, not Django admin only.

Main areas:

- Dashboard.
- Locations.
- Extensions.
- Phones.
- Phone book.
- Trunks/providers.
- Outbound routes.
- DID routes.
- IVRs.
- Audio prompts.
- Ring groups.
- Queues.
- Paging groups.
- Recordings.
- Exports.
- Audit log.
- Backups.
- Users/roles.
- API keys.

Bulk tooling:

- Forms for normal editing.
- CSV import/export templates for extensions, phones, DIDs, and speed dials.

Configuration versioning UI:

- Show export history per location.
- Show version number, checksum, export timestamp, exporting user, deployment status, and active PBX-reported version.
- Allow admins to download, deploy, or roll back to a recorded version.
- Show warnings when the active PBX version does not match the latest deployed or latest exported version.

## 16. Validation Rules

Warnings should be shown for:

- Missing provider secrets.
- Duplicate or suspicious DID routes.
- Invalid/incomplete phone metadata.
- Over-5 registration appearances per extension.
- Missing firmware/load names.
- Missing optional SMTP.
- Incomplete queue/IVR fallback destinations.

Hard export block:

- Missing emergency route or emergency caller ID for the location when any extension is allowed to dial emergency routes.

## 17. Security and Operational Constraints

- Portal is WARP or LAN only, not public internet.
- PostgreSQL database contains plaintext telecom secrets.
- Exported zips contain plaintext telecom secrets.
- Database, media, backup, exported zip, SSH private key, and deployment staging access must be restricted.
- All live PBX controls must be audited.
- Admins are responsible for legal/compliance requirements around emergency calling and call recording consent.

## 18. Backup Requirements

Central portal should support built-in backups containing:

- PostgreSQL database dump.
- Uploaded audio/media.
- Export history metadata.
- Configuration data.
- Audit logs, if practical.

Backups should be downloadable by admins and suitable for off-host storage.

## 19. API Requirements

The portal must expose authenticated APIs to get and update user information and control Channels.

Required API behavior:

- Admins can create, rotate, and revoke API keys.
- Only admins can create API keys.
- API keys must be scoped to a user or service identity.
- API key creation, rotation, revocation, and API-driven user updates must be audit logged.
- API callers can retrieve and update user info according to their permissions.
- Non-admin users cannot create API keys.

## 20. Configuration Versioning and Rollback

Every generated export must create an immutable config version record.

Each config version should record:

- Location.
- Version number.
- Exporting user.
- Export timestamp.
- Bundle checksum.
- Validation warnings.
- Emergency route status.
- Included Asterisk/TFTP file manifest.
- Deployment status.
- Deployment timestamp and operator.
- Asterisk reload result.
- Rollback source version, if applicable.

Rollback behavior:

- Admin selects a previous version.
- Portal deploys that previous version to the remote PBX host over SSH.
- Portal reloads Asterisk after the rollback deployment.
- PBX agent reports the newly active version back to the portal.
- Rollback action is audit logged.

PBX active-version reporting:

- Each deployed bundle includes a version marker file.
- The PBX agent reads the marker after deployment/reload.
- The agent reports active version, checksum, and timestamp to the portal.
- The dashboard highlights drift between latest exported, last deployed, and PBX-reported active versions.
